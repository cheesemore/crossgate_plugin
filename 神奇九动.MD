# 神奇九动（5 号位只动一次）

> 适用客户端：**魔力宝贝：序章**（`E:\cross\魔力宝贝：序章`）  
> 补丁工程来源：`crossgate_cursor` / `Crossgate客户端补丁` + `HotfixPatcher`  
> 序章版**不含宠物装备四孔**；下文互斥检查以「战斗多动类补丁」为主。

---

## 1. 功能说明

原版协议 **69**（`OnCommandPlayerCallback`）每回合把参战玩家 uid 依次写入 `BattleDataHolder.AcountList`，5 人队为：

```
P1 → P2 → P3 → P4 → P5   （每人 1 次，共 5 项）
```

客户端按队首顺序 `ChangeNextBattleRole()` → 选人 UI / 自动战斗发指令；队列为空后本回合指令阶段结束。

**神奇九动**把 5 人队队列扩为 **9 项**：

```
P1 → P2 → P3 → P4 → P1 → P2 → P3 → P4 → P5
```

| 玩家 | 出现次数 | 含义 |
|------|----------|------|
| P1～P4 | 各 **2** 次 | 人多一轮操作（二动） |
| P5（末位） | **1** 次 | 只动一次 |

N 人队通用：队列长度 `2×N−1`；前 `N−1` 人各 2 次，最后 1 人 1 次。

## 2. 实现原理（伪代码）

```csharp
// 协议 69 回调内

// 第一段：主循环（字典/数组等副作用仍每人只处理一次）
for (int i = 0; i < proto.Player.Count; i++)
{
    // PlayerActionMagics、AccountIndexDic、各 Flag 数组 … 照常

    if (i < proto.Player.Count - 1)
        AcountList.Add(proto.Player[i].Uid);   // 首轮只加到 P(N-1)
}

// 第二段：主循环结束后
for (int j = 0; j < proto.Player.Count - 1; j++)
    AcountList.Add(proto.Player[j].Uid);       // 再把 P1..P(N-1) 加一遍

AcountList.Add(proto.Player[Count - 1].Uid);   // 最后补 PN
```

补丁位置：

| 改动 | 类 / 方法 |
|------|-----------|
| 队列扩展 | `BattleProcesser.OnCommandPlayerCallback` |
| 二轮技能栏 | `PlayerActionMagics[uid]=true` 赋值处改为写 `false`（共 3 处，原地改 `ldc.i4.1`→`ldc.i4.0`） |

第二项避免：同一 uid 第二次进 `ChangeNextBattleRole` 时因「一动用过魔法/道具」导致二动 UI 被挡。

## 3. 与自动战斗的关系

- **69 不会被跳过**：无论手动还是自动，服务端每回合仍会推 69，客户端仍会建队。
- 区别在**消费队列**：
  - **手动**：每弹一次队首 → 一次选人 UI。
  - **自动**：`AutoFight()` 对当前 `CurrentAccount` 一次发完人+宠指令，然后 `ChangeNextBattleRole` 换下一个队首。
- **开局就自动**和**回合中切自动**都能走 9 项队列；自动没有 UI 轮次，看起来像「自动在跑」，不是跳过 69。

## 5. 互斥性检查（序章必读）

### 5.1 绝对不要与神奇九动同时打的战斗补丁

| 补丁 | 命令 / GUI | 冲突原因 |
|------|------------|----------|
| **战斗 Send 双发** | `battle-double-action-patch` | 在 `SendBattleCommond` 每条指令发 2 次；与九动「队列多轮」是两套多动机制，叠加会乱 |
| **强制二动 / 全量二动** | `battle-full-double-action-patch` | 手动双发 + 自动双发组合；同上 |
| **自动战斗双发** | `--battle-auto-double-action` | `AutoFight_PlayerAction` / `PetAction` 各执行 2 次 `Config[0]`；与九动队列叠加 |

### 5.2 同方法硬冲突：`PlayerActionMagics`

神奇九动与上述双发/二动补丁都会改 **`PlayerActionMagics` 赋值处的 IL**（同一 3 个方法）：

| 补丁 | 改法 |
|------|------|
| 神奇九动 | 强制写 `false`（二轮可开技能/道具栏） |
| 双发 / 二动 | 倾向保持 `true`（二动仍可用魔法/道具） |

**后打的会覆盖先打的，不能共存。**

### 5.3 序章不涉及：宠物装备四孔

完整版 `crossgate_cursor` 里曾有「四孔 + Send 双发」同 PE 多次写入崩溃，需组合补丁一次写入。

**序章无四孔选项**，无需考虑「九动 vs 四孔」；若以后单独加四孔 UI 补丁，与九动**功能上不冲突**（改 `PetEquipChildPanel` vs 改 `OnCommandPlayerCallback`），但**组合稳定性需单独实测**，当前工具仍按「战斗多动类三选一」设计。

### 5.4 脚本层互斥（`apply_combo_patch.py`）

```text
宠物四孔 + 战斗双发 + 神奇九动 → 三者只能选一个（exclusive_count > 1 报错）
```

序章 GUI 已去掉四孔、双发、自动双发；只勾神奇九动时 `exclusive_count = 1`，通过检查。

### 5.5 可安全叠加（已验证思路）

与九动改动的类**无交集**：

- `battle-nav-show-patch` — 挂机按钮  
- `vip-timescale-patch` — 战斗/心跳倍速  
- `battle-longpress-patch` — 长按详情  
- `map-sprint-speed-patch` — 地图跑速  
- `skill-effect-speed-patch` — 技能特效帧率  
- `mige-panel-patch` — 秘阁顶栏面板

## 8. 技术备注

- 方法体变大时走 `.text` slack **追加** + 更新 MethodDef RVA；文件总大小保持 **6,355,968**。  
- `IlSerializer` 须正确处理双字节 opcode（`Ldloc_S` 等），否则 IL 错位崩溃。  
- 次轮 `List.Add` 栈顺序须与原版一致：`get_Uid → stloc → ldsfld AcountList → ldloc uid → Add`。  
- 最后补 PN 时 `get_Item` 须先算 `Count-1` 再重新 `get_Player` + `get_Item`，不能 `get_Count` 后直接 `get_Item`。
